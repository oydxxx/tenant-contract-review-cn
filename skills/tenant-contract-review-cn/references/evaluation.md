# 合成前向评测与人工复核

发布前执行：

```sh
python3 scripts/run_forward_evals.py --cases tests/forward-evals --output build/eval-report.json
```

评测只读取 `tests/forward-evals/` 内明确标记 `synthetic: true` 的 JSON 状态。它不调用模型、不传入提示词、不读取真实合同，也不把某个预期中文结论交给评测代理。每个案例只声明需要成立的机器规则，由评测器独立检查；任何硬门禁失败都会使命令以非零状态退出。

## 覆盖与硬门禁

当前套件以 14 个独立案例覆盖 AE1–AE14，并额外检查：宿主能力不同导致准入不同、替换页面、规则到期、能力变化使下游检查点失效、危险附件与提示注入被阻断，以及无证据时不能给出正向或劝退结论。

`build/eval-report.json` 固定包含：

- `report_version`、案例 ID 和 AE 覆盖；
- 硬门禁总数、通过率、失败记录和失败分类；
- 是否仍需人工复核及对应案例；
- 明确的“仅合成数据、确定性、未使用模型”边界。

报告不含时间戳，使用相同案例会产生相同内容。修改规则、案例结构或评测器语义时，维护者必须提升 `report_version`，重新运行并随候选发布留存报告。

## 人工复核不能省略

100% 硬门禁只证明编码的规则未被破坏，不能证明法律适用、证据充分性、中文谈判表达或对具体当事人的建议正确。发布审核必须逐案复核报告标注的人工项目，尤其是：红线证据链、地方规则的适用边界、谈判文字与让步边界、终稿变更比较，以及对隐私与宿主能力声明是否如实披露。

只有自动检查通过、人工复核完成、提交与许可证一致，且首次发布的远程账号、仓库名、公开可见性、标签和 Release 由维护者明确确认后，才能进行 GitHub 发布。
